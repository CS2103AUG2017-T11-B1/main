Project: Directory of Contacts (DoC)

DoC is a desktop contact and event managing application. It is built as an enhancement on top of the original Addressbook-level4 by Team SE-EDU, which can be found here: https://github.com/se-edu/addressbook-level4. This project was built as part of the software engineering module CS2103T offered by the School of Computing in the National University of Singapore.

Event features

You can manage your events through the following commands:

Adding an event: addE

Adds an event to DoC.
Format: addE h/HEADER de/DESCRIPTION d/DATE

Quick note:

The date must a valid date with the format: year-month-day (yyyy-MM-dd).

Examples:

  • addE h/Meeting de/Business meeting at in the afternoon at 2pm d/2017-11-20

  • addE h/Birthday de/John Doe’s 21st bd celebration d/2018-02-05

Visual guide:

Your newly created event will be listed in the event panel and will be sorted based on its date. Thus, the earliest event will be displayed at the top of the panel. As shown in the guide above, the yellow panel indicates that an event upcoming in 2 or less days, the green panel indicates that an event that have yet to come (more than 2 days) and the red panel indicates that the event is already overdue.

Editing an event : editE

Edits an existing event in DoC.
Format: editE INDEX [h/HEADER] [de/DESCRIPTION] [d/DATE]

Quick notes:

  • Edits the event at the specified INDEX. The index refers to the index number shown in the last event listing. The index must be a positive integer 1, 2, 3, …​

  • At least one of the optional fields must be provided.

  • The date must a valid date with the format: year-month-day (yyyy-MM-dd).

  • Existing values will be updated to the input values.

Example:

editE 3 h/Birthday party d/2017-10-30
Edits the header and date of the 1st event to be Birthday party and 2017-10-30 respectively.

Visual guide:

According to the image above, if you want to edit an Event at index 3 by changing the header and date, the edited event will be automatically sorted in the Events panel based on the new date.

Deleting an event : deleteE

Deletes the specified event from DoC.
Format: deleteE INDEX

Quick notes:

  • Deletes the event at the specified INDEX.

  • The index refers to the index number shown in the most recent listing.

  • The index must be a positive integer 1, 2, 3, …​

Example:

deleteE 3
Deletes the 3rd event in DoC.

Visual guide:

As shown on the image above, if you want to delete an overdue Event, simply key in the deleteE command stating the index and the event will be removed from DoC.

End of Extract

Justification

The added layer of event management helps to improve the user’s experience. This feature allows DoC to manage two of the most fundamental aspects of the user’s daily life. That is, to keep track both contacts and tasks and have access to either within the same platform. For busy individuals, this eliminates the trouble of switching between their contact list and calendar as DoC display both scenes in one and provides a better notification system. All events are sorted based on their allocated dates and sets a color to every event’s background and displays a different color based on the urgency of an event. This added improvement aims to provide convenience when it comes to planning upcoming tasks/events.

Event class mechanism

Summary

The Event class, which is located inside Model, is implemented with similar logic as the Person class. We have introduced three commands that modify the address book: addE, editE and deleteE, which extend UndoableCommand.

An Event consists of three sub-components: Header, Desc and EventDate. Commands such as clear, undo and redo can be used to alter events in the list as they deal with code that directly modifies the address book.

The EventPanel is incorporated into the MainWindow of the DoC to display all the events inside the internal list using ObservableList<ReadOnlyEvent>. This process will be explained later on under the section Event card.

Event methods

As stated in the summary, the Event class contains three main methods: addE, editE and deleteE. The execution flow is similar for all three methods on a higher level. Exceptions

When the user inputs an event command with its parameters, the validity of the command word is checked inside the AddressBookParser. Next, the presence of the parameter prefixes is checked inside AddEventCommandParser. An appropriate ParseException will be thrown if the command word or prefixes are incorrect.

Before modifying the list of events inside the address book, the system may throw exceptions due to some invalid parameters. Here are the possible exceptions:

  • EventNotFoundException: This exception can be thrown by editE and deleteE command when the an invalid event index is provided by the user. The index needs to be within the size of the event list at the current state of the address book.

  • DuplicateEventException: This exception can be thrown by addE and editE command. The system will first create an Event object with the input parameters, compare the object to all events residing in the event list and throw this exception if there is a duplicate found.

  • DateTimeParseException: This exception can be thrown by addE and editE commands when the input EventDate parameter is not a valid date. The date’s format and validity are checked inside the Event class like this:

public EventDate(String eventDate) throws IllegalValueException {
    DateTimeFormatter formatter = DateTimeFormatter.ofPattern("yyyy-mm-dd")
                    .withResolverStyle(ResolverStyle.STRICT);
    try {
        eventLocalDate = LocalDate.parse(eventDate, formatter);
    } catch (DateTimeParseException ex) {
        // ... throw message ...
    }
}

Event card

The EventCard class extends UiPart<Region> to represent a distinct part of the UI. The object properties of every event is assigned to a label held by an EventCard. The graphic scene is then constructed with the appropriate FXML files created to support the display of all events.

Design implementation

When the user starts the MainApp, the system calls the UiManager to create a new MainWindow and fills it with EventListPanel and other components. The displayed events are created by UniqueEventList and the binding of individual UI elements to the EventCard ensures that any changes to the parameter will be displayed in the EventListPanel. The events on the EventListPanel are sorted based on their respective EventDate. An event that is coming soon will appear at the top and events with later dates will be further down the list. Expired events will be pushed to the bottom of the list. This sorting algorithm is introduced in EventDateComparator which implements Comparator<Event>.

Design considerations

Aspect (future enhancement): Implementation of adding/tagging of contacts into an Event using a Person list parameter

Chosen Implementation:
Add by the index of contact shown in the PersonListPanel.
Pros:
Only check for validity of index has to be done by system. Better performance compared to Alternative.
Cons:
Requires user to refer to the PersonListPanel before executing command to add contact into event’s Person list.

Alternative:
Add by the name of contact in the Person list.
Pros:
Easier for users to add using names, do not need to refer to the Person list.
Cons:
Check through the list to check if the contact’s name exist in the current address book has to be done by system.

Enhancement Proposed: Tagging of contacts into events

The ability to tag contacts to an event can further enhance user’s experience with our product. Both features compliment one another as users can view an event and check for its involved personnel so that they can contact them by using the contact list in DoC. With an email feature created by one of our developers, the whole process tie in together when a user is able schedule an event, tags the intended contacts and email them with minimal command words.
